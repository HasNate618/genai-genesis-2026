# Implementation Questions

## Architecture & Infrastructure
1. What vector database should be used for the Semantic Memory Layer? (e.g., ChromaDB, Pinecone, Weaviate, Qdrant)

Answer: Core databasing should be done using the [moorcheh python sdk](https://github.com/moorcheh-ai/moorcheh-python-sdk).

2. Should the Python backend run locally or be deployed to a server? If local, should it use local LLMs (Ollama) or cloud APIs?

Answer: We will use cloud models, but the backend will run locally.

3. What LLM providers/models should be used for each agent role?

Answer: For sake of simplicity, all agent roles will be configured to use the same model; we will use Gemini 3.1 pro.

https://github.com/moorcheh-ai/moorcheh-python-sdk

## VSCode Extension
4. Should the extension connect to a local FastAPI server or use a built-in agent runtime?

Answer: a Local FastAPI server should be used.

5. What specific tools should be injectable into coder agents? (file system, git, terminal, etc.)

Answer: Use the following permissions for the actual code-writing agents:
{
  write: true
  edit: true
  bash: true
  read: true
  grep: true
  glob: true
}

6. How should agent isolation be implemented? (separate directories, containers, or something else?)

Answer: Seperate directories should be implemented for each agent.

## Agent Configuration
7. How many concurrent coder agents should be supported by default?

Answer: Three paralell agents should be supported by default. Note that the actual *coding* agents (not planning, coordination, etc) will run in paralell with each other. 

8. What is the conflict score threshold for requiring task reassignment?

Answer: The conflict score threshold is 20% semantic similarity between task summaries, Using the task summaries generated by the task coordinator agent. The equation for calculating the score is:

```
riskScore = [(numOfSharedDependencies * 0.2) + (percentageOfTaskSimilarity * 0.5) + (isSameCategoryTask * 0.3)]
```

9. Should agents be configurable by users or use fixed role definitions?

Answer: Agents should have fixed role definitions. The user will not be able to configure the workflow process. 

## Workflow Details
10. Should the semantic memory persist across sessions, or only within a workflow?

Answer: Semantic memory should persist only within single workflows.

11. How should merge conflicts be handled - automated resolution attempts or human-assisted?

Answer: Merge conflicts should be handled automatically. As step 7 of the workflow states, if the merge agent has difficulty, the workflow will return to step 4.

12. What types of QA tests should the tester agent run? (unit tests, integration tests, custom scripts?)

Answer: The QA agent will only run demos of the generated code. If the demo succeeds, it is funished; if not, return to step 5.

## Human Approval
13. Should approval be blocking (synchronous) or can it be asynchronous/notifications?

Answer: Approval requires direct intervention from the user. 

## Data & Storage
14. Where should workflow history and agent outputs be stored? (local files, database, both?)

Answer: Workflow history and agent outputs should be stored via the moorcheh python sdk. If moorcheh does not fit the task, SQLite can be used instead.

15. Should there be a notion of "projects" or "workspaces" to organize multiple workflows?

Answer: All current and previous workflows should be organized as "projects".
